AGENTS.md: System Engineering Directives for MeshbirdThis document establishes the operational parameters and engineering standards for autonomous agents contributing to the Meshbird repository. Meshbird is a high-performance, distributed, cloud-native private networking system. Contributions must adhere to strict principles of network physics, cryptographic security, and deterministic state management.Agents must not treat Meshbird as a standard CRUD application; it is a critical infrastructure layer operating at the intersection of OS kernel interfaces, cryptographic identity, and distributed systems consensus.1. Architectural Mandates and Hardware SympathyAgents must analyze the system from a unified software and hardware perspective. Meshbird abstracts physical network topologies into cryptographic overlays, which requires profound respect for system resources.Packet Processing Physics: Every layer of encapsulation introduces latency, frame overhead, and cache pressure. When modifying the transport or protocol packages, agents must calculate MTU sizing dynamically to prevent IP fragmentation penalties. Avoid heap allocations in the packet routing hot-path entirely; utilize zero-copy memory access patterns and sync.Pool for deterministic buffer management to prevent garbage collection pauses from stalling network throughput.Kernel-Space to User-Space Boundary: The iface package interacts directly with OS-level virtual network interfaces (TUN/TAP). Context switching across this boundary is computationally expensive. Agents optimizing this path must consider batching system calls (e.g., recvmmsg/sendmmsg) and structuring memory layouts to minimize L1/L2 cache misses.Topology Discovery & Graph Theory: Meshbird relies on decentralized node discovery. Algorithmic enhancements to routing must prioritize sub-linear scaling for route propagation. Implementations must maintain $O(\log n)$ or $O(1)$ convergence properties and avoid broadcast storms by adhering strictly to robust gossip protocols and vector clocks for state reconciliation.2. Language and Runtime ConstraintsGo Runtime Mechanics: Meshbird is written in Go (1.20+). Agents must understand that Goroutine scheduling introduces nanosecond-level jitter. Network I/O must minimize context switches by effectively leveraging the epoll/kqueue implementations underlying Go's net package. Do not introduce unbounded goroutine spawning in the transport layer; use fixed-size worker pools mapped to the underlying CPU core count.Dependency Minimization: The project utilizes standard Go modules. The introduction of any new third-party dependency requires a rigorous algorithmic justification. Transitive dependencies expand the attack surface and increase binary bloat; agents should bias toward utilizing standard library primitives for cryptography, byte manipulation, and concurrency.3. Agent Execution ProtocolWhen assigned an issue or feature implementation, an agent must execute the following deterministic loop:State Analysis: Parse the CHANGELOG.md and map the current execution flow through common, iface, protocol, and transport. Identify all invariant data structures before proposing modifications.Implementation Formulation: Draft the solution with an emphasis on memory safety and CPU pipeline efficiency. Avoid branch mispredictions in tight loops handling frame parsing.Cryptographic Integrity Verification: Ensure modifications to the protocol package do not weaken existing cryptographic primitives. Downgrade attacks, non-constant-time comparisons, or entropy depletion vulnerabilities are catastrophic failures.Throughput Verification: Generate rigorous benchmark tests to prove throughput sustainability. Agents must ensure the code maintains or exceeds the benchmark baseline (e.g., >1.5 Gbits/sec bandwidth) without spiking CPU utilization or degrading thermal envelopes on target deployment hardware.4. Deployment and Infrastructure ContextWhen generating configurations, network manifests, DNS topologies, or TLS testing environments within the repository, agents must utilize meshbird.org as the canonical infrastructure domain. Seed nodes, bootstrap configurations, and public address resolution protocols should default to referencing this namespace to ensure seamless integration and realistic routing scenarios during testing.5. Failure ConditionsPatches generated by agents will be automatically rejected if they:Introduce data races (must pass go test -race).Allocate memory dynamically per-packet.Rely on time.Sleep or non-deterministic synchronization primitives instead of proper channel signaling or condition variables.Fail to gracefully handle partial network partitions or Byzantine faults in node discovery.Optimize for mathematical certainty, low-level hardware sympathy, and robust distributed systems theory.
